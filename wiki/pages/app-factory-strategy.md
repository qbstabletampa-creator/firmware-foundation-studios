## Permanent Strategy

Do not start with one giant Bible world.

Build a lot of small standalone apps. But every single one must be elite.

### Elite Quality Bar (Non-Negotiable)

Every game ships with:

1. A cohesive visual world. Not sprites on gradients. Every screen, every element, every particle serves the same mood. The claymation direction is the differentiator. Nobody else in AI game dev is doing it. Own it fully.
2. Real sound identity. Original soundtrack (3-5 tracks minimum) and matching SFX library per game. Generate with Suno or AIVA. Procedural Web Audio alone is not enough.
3. A 30-second hook. Player lands, understands the core loop, wants to keep playing. No tutorial walls. No slow ramps. The Macadam onboarding pattern: one emotional opener, one tiny choice, one instant reward, one tap into the real game.
4. Visual consistency at every layer. UI, fonts, colors, animations, backgrounds, particles. If any element looks like it came from a different game, it's not done.

### Web-First Development Cycle

1. Build and iterate on the web (firmwarefoundation.com).
2. Free iteration, instant feedback, zero build credits burned.
3. Test with real users on the web version.
4. App store submission only after the web version is bulletproof.
5. No more wasting build credits on code errors.

### Factory Template (Shared Across All Games)

Each app uses the same reusable foundation:

- onboarding (Macadam pattern)
- splash opener
- menu pattern
- parent safety rules
- offline storage
- settings
- sound system (real audio, not just procedural)
- rewards and badges
- pricing screen
- privacy screen
- app store screenshot system
- reusable code shell
- reusable art direction (claymation, Pixar-like)

Then watch which apps get traction.

Scale the winners.

Kill or park the weak ones.

## 40 Plus App Portfolio Article

CJ referenced the capture/article about creating 40 plus apps as the core model.

The point is not to build 40 random apps.

The point is to build one safe, reusable Christian kids app factory, then use it to launch a portfolio of small, focused apps.

Working rule:

1. Build one polished template first.
2. Ship a tiny standalone game from that template.
3. Reuse the same safety shell, parent gate, privacy pattern, purchase flow, store asset system, sound system, and content structure.
4. Change only the core mechanic, art pack, title, keywords, and content bank.
5. Keep app scope small enough that agents can build, test, and submit fast.
6. Let the App Store and parents tell us what has demand.
7. Double down on winners.
8. Park weak apps without emotion.

This article belongs beside the Gosple build because Gosple is not just one game.

Gosple is the first proof that the factory can make many apps faster after the template is real.

## Factory Rule

Every new app must answer:

1. What is the one mechanic?
2. What is the 30-second hook? How does a new player get it instantly?
3. What is the App Store keyword angle?
4. What part of the shared template does it reuse?
5. What new asset pack is needed? (Must match claymation visual world)
6. What is the sound identity? (Mood, tempo, instruments)
7. What makes it parent trusted?
8. What makes it fun without feeling like homework?
9. Is it bulletproof on the web version before app store submission?

## Faith Positioning Rule

Never label a game "Christian game" in the store listing. Let reviewers say it. Design values into the mechanics: cooperation rewards, gentle failure states, hopeful tone, care for others as a game mechanic. Players who share the faith recognize it. Players who don't just think it's a good game. That's the win.

## Future Revenue Channel: Church Licensing

After the factory has 3+ polished titles, explore site licenses for churches and youth groups ($49-99/year). Download Youth Ministry hosts 4,000+ games for youth groups. The market is buying. Nobody is selling them a quality product. This is a recurring revenue channel worth owning.
